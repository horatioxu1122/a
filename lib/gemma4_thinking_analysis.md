# How Gemma 4 Thinking Works: Binary and System Level Analysis

**Date:** 2026-04-04
**Method:** Binary string extraction from ollama v0.20.2, modelfile inspection, API testing

## Architecture

Thinking in Gemma 4 is a three-layer system: model weights, token protocol, and ollama parser.

### Layer 1: Model Weights (trained behavior)

Gemma 4 was **trained** to generate `<think>...</think>` blocks before answering. This is not a system prompt injection — it's learned behavior in the transformer weights, similar to how the model learned to generate English. When the model sees the chat template `<start_of_turn>model\n`, it has a high probability of emitting `<think>` as the first token because that's what the training data contained.

The thinking tokens are generated by the same weights, same attention heads, same FFN layers as the answer tokens. There is no separate "thinking module" — the model simply generates tokens in `<think>` tags first, then content after `</think>`. The distinction is purely in the token sequence, not the architecture.

**Implication for pruning:** You cannot selectively prune "thinking layers" because they don't exist. Every layer participates in both thinking and answering. Pruning a layer degrades both equally. The `layer_output_scale` values are task-agnostic — a layer with scale 0.036 contributes 3.6% to ALL generation, thinking included.

### Layer 2: Token Protocol

The chat template from the GGUF metadata:

```
<start_of_turn>user
{user message}
<end_of_turn>
<start_of_turn>model
{model generates here}
```

The model then generates:
```
<think>
{reasoning tokens — 200-800 tokens typically}
</think>
{actual answer}
<end_of_turn>
```

The transition sequence found in the binary: `></think><start_of_turn>developer` indicates that after thinking completes, the model can also enter a "developer" turn for tool-use scenarios.

There is **no system prompt** for thinking. The modelfile shows:
```
TEMPLATE {{ .Prompt }}
RENDERER gemma4
PARSER gemma4
```

No SYSTEM field. The `{{ .Prompt }}` template is expanded by the renderer into the Gemma chat format. The thinking behavior comes from the model weights responding to this format.

However, the string `Enable deep thinking subroutine.` was found in the binary — this appears to be an optional system prompt that ollama can inject to encourage thinking. It is not used by default.

### Layer 3: Ollama Parser (Go code)

Ollama's parser layer handles the `<think>` tokens. From the binary:

```
parsers.gemma4Event              — base event type
parsers.gemma4EventThinkingContent — thinking token event
parsers.gemma4EventContent       — content token event  
parsers.gemma4EventToolCall      — tool call event
thinking.Parser                  — separate thinking parser stage
thinking.thinkingState           — state machine for think/no-think
```

The parser is a state machine:
1. Model generates token
2. If token is `<think>`: enter thinking state, route to `"thinking"` JSON field
3. If token is `</think>`: exit thinking state
4. Otherwise: route to `"content"`/`"response"` JSON field

In the streaming API response:
```json
{"model":"gemma4:e2b","thinking":"I need to calculate...","done":false}
{"model":"gemma4:e2b","thinking":"347 * 28 = ...","done":false}
{"model":"gemma4:e2b","response":"9716","done":false}
{"model":"gemma4:e2b","response":"","done":true}
```

### Control Mechanism

The `think` parameter controls thinking at generation time:

| Value | Effect | Mechanism |
|-------|--------|-----------|
| `true` (default) | Full thinking | Model generates freely |
| `false` | No thinking | `<think>` token suppressed at logit level |
| `"low"` | Brief thinking | Thinking budget constrained |
| `"medium"` | Moderate thinking | — |
| `"high"` | Extended thinking | Maximum thinking budget |

From the binary: `think must be a boolean or string ("high", "medium", "low", true, or false)`

When `think=false`: ollama sets the `/nothink` flag (found as `/nothinkL` in binary). This likely applies a logit bias of -infinity to the `<think>` token ID, making it impossible for the model to generate. The model then generates content directly.

There is also a `gemma4-no-thinking` variant string in the binary — suggesting ollama may have a built-in model alias that defaults to no-think mode.

## Performance Impact

Measured on RTX 4080 12GB, gemma4:e2b:

| Mode | Tokens before answer | Time overhead | Quality |
|------|---------------------|---------------|---------|
| `think=false` | 0 | 0 | 90% (Tier 1) |
| `think=true` | 200-800 | 1-4 seconds | 100% (Tier 2) |

The thinking tokens are generated at the same speed as answer tokens (182 tok/s on E2B). The overhead is purely the extra token count. At 500 thinking tokens average, that's 2.7 seconds per question.

## Why E2B Beats E4B With Thinking

Benchmark results: E2B scored 100% vs E4B's 80% on advanced reasoning with thinking enabled.

Hypothesis: **Thinking quality depends on the thinking token budget, not the model size.** Both models think for similar token counts (200-800), but E2B processes those tokens 1.5x faster (182 vs 119 tok/s). With a fixed max_tokens of 1024, E2B has more effective tokens for the answer after thinking. E4B may be spending more tokens on verbose thinking and running out of budget for the answer.

Alternative hypothesis: E4B's architectural differences (shared KV layers, bottleneck gating) may interfere with the thinking mechanism in ways that don't affect direct answers. The 18 shared KV layers in E4B mean thinking tokens share attention state with the eventual answer in a way that E2B's independent KV caches don't. This could cause thinking-answer interference.

## Implications for Compression

1. **Thinking is the equalizer.** A smaller model that thinks well beats a larger model that thinks poorly. Compression should be measured with thinking enabled, not disabled.

2. **Thinking and answering share weights.** There is no separate thinking circuit to preserve or discard. Pruning affects both uniformly.

3. **The `layer_output_scale` tells you how much each layer contributes to ALL generation.** A layer with scale 0.036 (blk.59) contributes 3.6% to thinking AND answering. Removing it degrades both by the same amount.

4. **Token budget matters more than parameter count.** When benchmarking pruned models, fix the thinking token budget (e.g., max 512 thinking tokens) to isolate the effect of pruning from the effect of thinking verbosity.

5. **`think=false` is the correct mode for fast quality probes during pruning.** It tests the core weights without the confound of variable thinking length. `think=true` is for final validation only.
